# 使用 AI 开发助手开发 JitAI 元素

AI 开发助手是面向开发者的内置助手，用于在 IDE 中通过自然语言创建、修改和修复 JitAI 元素。它不是普通业务聊天 Agent，而是开发工作台中的协作开发助手，适合处理模型、服务、页面、Agent、Skill、Shell、事件、任务和数据库配置等开发任务。

开发助手的核心价值是把“理解需求、定位元素、生成改动、预览差异、应用变更、构建刷新”串成一个开发闭环。开发者仍然负责确认需求、审阅变更和验证结果。

## 打开 AI 开发助手 {#open-ai-development-assistant}

在 IDE 中打开应用后，可以从开发界面的 AI 开发助手入口进入。若当前已经打开某个元素，开发助手会自动带上当前元素上下文，便于你直接提出修改要求。

例如当前打开的是 `models.sales.CustomerOrder`，可以直接说：

```text
给当前订单模型增加一个“客户等级”字段，并同步补充列表页展示。
```

如果没有打开具体元素，也可以直接说明要开发的对象：

```text
创建一个客户回访记录模型，并生成配套的新增、编辑和列表页面。
```

## 首次使用前配置开发模型 {#configure-developer-model}

首次使用 AI 开发助手时，需要配置开发助手使用的大模型。这个模型需要支持 Function Calling，否则开发助手无法稳定完成元素查询、差异预览、应用变更和构建等操作。

配置时建议：

- 选择上下文长度较大、代码生成能力稳定的模型。
- 确认模型支持 Function Calling。
- 把 API Key 等敏感信息保存为环境变量或模型服务配置，不要写入对话。
- 如果模型经常输出不完整代码，优先降低随机性参数，并选择更稳定的代码模型。

开发模型配置完成后，同一应用后续会复用该配置。

## 开发助手能做什么 {#capabilities}

AI 开发助手适合处理以下开发任务：

| 任务类型 | 可以怎么用 |
| --- | --- |
| 数据模型 | 创建模型、补充字段、调整字段标题、增加索引或关联关系 |
| 服务函数 | 新建服务、补充业务函数、修复参数和返回结构 |
| 页面 | 创建列表页、表单页、详情页，调整页面组件和交互 |
| Agent | 创建或修改 Agent 配置、提示词、工具、知识库和文件空间 |
| Skill | 创建可复用 Skill，补充说明、参考资料和任务规范 |
| Shell / 门户 | 创建或调整门户、菜单和页面入口 |
| 事件 / 任务 | 创建事件处理逻辑、定时任务或后台处理流程 |
| 数据库配置 | 辅助梳理数据库连接和模型映射需求 |

如果一个需求跨多个元素，开发助手会先拆分任务，确认每个元素的目标和边界，再逐个处理。

## 推荐的协作流程 {#recommended-workflow}

使用 AI 开发助手时，建议按下面流程协作：

1. **说明目标**：用业务语言说明你要创建或修改什么。
2. **确认范围**：如果涉及多个元素，先让开发助手列出元素清单和修改计划。
3. **逐个处理**：一次聚焦一个当前元素，避免多个改动混在一起难以审阅。
4. **审阅差异**：开发助手生成改动后，先查看差异预览。
5. **应用变更**：确认无误后再应用到源码。
6. **构建刷新**：让开发助手构建相关元素，并刷新 IDE 中的元素状态。
7. **运行验证**：打开页面、执行函数或运行 Agent，验证结果是否符合预期。

对于复杂需求，可以先让开发助手生成任务拆解：

```text
我要做一个售后工单管理功能。请先列出需要创建或修改的模型、服务函数、页面和权限点，不要直接改代码。
```

确认拆解后，再让它逐项执行。

## 差异预览和变更应用 {#diff-review-and-apply}

AI 开发助手默认以差异预览的方式提交代码改动。开发者应先审阅新增文件、修改位置和关键逻辑，再决定是否应用。

差异预览适合：

- 新建或修改数据模型、服务函数、页面和 Agent。
- 检查字段名称、函数参数、页面绑定和提示词是否符合预期。
- 发现开发助手误改了不相关内容时及时打断。

只有在你明确表示“跳过审阅，直接应用”时，开发助手才应直接应用变更。日常开发建议始终保留差异审阅步骤，尤其是涉及数据删除、权限、业务状态流转和外部接口调用的改动。

## 构建、刷新和验证 {#build-refresh-and-verify}

应用变更后，开发助手可以继续执行构建和刷新：

- **构建元素**：适合只修改了单个模型、服务、页面、Agent 或 Skill 的场景。
- **构建应用**：适合跨多个元素、依赖关系变化或应用级配置变化的场景。
- **刷新元素**：让 IDE 重新加载元素，确保可视化编辑器和运行态看到最新内容。

构建完成不代表业务已经验证通过。开发者还需要根据元素类型做对应检查：

- 模型：字段、索引、关联关系、数据权限是否正确。
- 服务函数：参数、返回值、异常处理和权限是否符合预期。
- 页面：组件展示、事件绑定、表单提交和数据刷新是否正常。
- Agent：模型、提示词、工具、知识库、文件空间和预置问题是否符合任务边界。
- Skill：说明是否清楚，参考资料是否完整，是否能被目标 Agent 正确使用。

## 提问方式示例 {#prompt-examples}

清晰的开发请求能明显提高结果质量。

| 目标 | 示例 |
| --- | --- |
| 修改当前元素 | `当前页面的表格增加“客户等级”列，并支持按客户等级筛选。` |
| 新建模型和页面 | `创建客户回访记录模型，字段包括客户、回访时间、回访方式、回访结论，并生成列表页和编辑页。` |
| 修复服务函数 | `检查当前服务函数为什么保存订单时报错，保留现有业务逻辑，只修复参数和空值处理。` |
| 扩展 Agent | `给当前 Agent 增加订单知识库，并补充提示词：涉及订单政策时先查知识库。` |
| 创建 Skill | `基于客户订单模型创建问数 Skill，覆盖订单明细查询、按客户统计和金额 TopN。` |

不建议一开始就提出过大的请求，例如“帮我做完整 CRM”。可以先让开发助手拆分模块，再分阶段实施。

## 使用边界和注意事项 {#boundaries-and-notes}

- AI 开发助手面向开发者使用，不应开放给普通业务用户作为日常业务 Agent。
- 生成的变更需要开发者审阅，尤其是数据模型、权限、删除操作、外部接口和批量更新逻辑。
- 普通业务 Agent 不应绑定源码工作空间，也不应具备修改应用源码的能力。
- 涉及多个元素时，优先让开发助手先列计划，再逐项执行。
- 如果需要新增依赖，要求开发助手说明依赖用途，并在构建后验证相关元素。
- 如果开发助手的结果偏离目标，应直接指出“只保留某部分，撤回某部分，重新生成差异”。

AI 开发助手适合提高开发效率，但它不能替代开发者对业务规则、权限边界和上线质量的判断。
